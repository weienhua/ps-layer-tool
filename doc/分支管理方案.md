# GitHub Flow 分支管理方案

## 分支结构

```
master (生产分支，始终可部署)
  │
  ├── feature/* (功能分支)
  ├── fix/* (修复分支)
  └── docs/* (文档分支)
```

## 分支命名规范

| 类型 | 命名格式 | 示例 |
|------|----------|------|
| 新功能 | `feature/简短描述` | `feature/card-collapse` |
| Bug修复 | `fix/简短描述` | `fix/color-display` |
| 文档更新 | `docs/简短描述` | `docs/update-readme` |

## 提交信息规范

```
feat: 新功能
fix: 修复bug
docs: 文档更新
style: 代码格式调整
refactor: 重构
test: 测试相关
chore: 构建/工具相关
release: 版本发布
```

## 工作流程

### 1. 开发新功能

```bash
# 1. 确保 master 是最新的
git checkout master
git pull origin master

# 2. 创建功能分支
git checkout -b feature/功能名

# 3. 开发并提交
git add .
git commit -m "feat: 添加xxx功能"

# 4. 推送到远程（可选，备份用）
git push origin feature/功能名

# 5. 开发完成后，合并到 master
git checkout master
git merge feature/功能名

# 6. 推送到远程
git push origin master

# 7. 删除功能分支
git branch -d feature/功能名
git push origin --delete feature/功能名
```

### 2. 修复 Bug

```bash
# 1. 创建修复分支
git checkout master
git checkout -b fix/问题描述

# 2. 修复并提交
git add .
git commit -m "fix: 修复xxx问题"

# 3. 合并到 master
git checkout master
git merge fix/问题描述
git push origin master

# 4. 删除修复分支
git branch -d fix/问题描述
```

### 3. 版本发布

```bash
# 1. 更新版本号
npm run release -- 1.0.4

# 2. 推送 tag 触发自动构建
git push origin master --tags
```

## 版本发布流程

```
1. 开发完成
   ↓
2. 运行 npm run release -- x.x.x
   ↓
3. 自动更新 package.json 和 doc/使用文档.md
   ↓
4. 自动创建 commit 和 tag
   ↓
5. git push origin master --tags
   ↓
6. GitHub Actions 自动构建并发布
```

## 注意事项

1. **功能分支要小**：每个分支只做一个功能，完成后立即合并
2. **频繁同步**：如果开发时间较长，定期从 master 合并最新代码
3. **测试后再合并**：确保功能正常再合并到 master
4. **master 始终可部署**：任何时候 master 的代码都应该是可用的

## 分支生命周期

```
创建 → 开发 → 测试 → 合并 → 删除
  │      │      │      │      │
  │      │      │      │      └── 合并后立即删除
  │      │      │      └── 合并到 master
  │      │      └── 确保功能正常
  │      └── 在功能分支上开发
  └── 从 master 创建
```

## 常用命令

```bash
# 查看所有分支
git branch -a

# 查看当前分支
git branch

# 切换分支
git checkout 分支名

# 创建并切换分支
git checkout -b 分支名

# 删除本地分支
git branch -d 分支名

# 删除远程分支
git push origin --delete 分支名

# 查看分支合并状态
git branch --merged
git branch --no-merged
```
